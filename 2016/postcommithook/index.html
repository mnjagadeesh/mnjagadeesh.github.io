Build a product with git Hook !!

I noticed that few software product companies have very complex product building system. For experienced engineers spent good amount of time understanding the system. Based on system complexity, they will be quite productive for few weeks or in some cases months. Initial experience will not be smooth for them.

On the other hand, companies have to spend resource on creating and maintaining document about building product.

Team responsible for maintaining find difficulty updating a system, making modification, adding any new feature because it involves, sometime extensive, training and documentation.

I was thinking to solve this problem. My initial idea was to choose popular tool that is almost known to everyone.

My obvious first choice was to make use of Jenkins. But to run Jenkins, we need to install jre and spare a network port. Again, not sure if everyone can use Jenkins. I have seen few folks who are reluctant to use it.

During this time, I was fine-tuning git hooks. I, personally, use hooks for mutt, emacs, version control systems. So I got an idea if git post-checkout hook can be used for to build the product. Idea is automating all tedious manual steps.

To experiment this idea I chose a small project built using [buildout][http://docs.buildout.org/]. It is a just simple CRUD application built using python and zope and MSSQL as backend database.

To set up this application little complex, because it makes use of various third party components like RabbitMQ, Memcache and many other python modules. The buildout configuration is slightly complex and takes about 50 minutes to complete build. Configuration involves setting network port, database info and many other values.

To begin my testing, I automated manual steps with a python script in post-checkout, updated ~/.gitconfig to make hooks effective. Cloned repo of product. After successful Cloning, default branch was checkedout. At this point automagically, post-commit hook started executing. It did update buildout configuration, executed buildout successfully. Product installation is successful, I could see login page.

Since all steps were executed in machine time, complete build took considerably less time.

Next step is start considering different use cases.

For a new engineer, it takes good amount of time to collect required information, configure and setting up product. If it breaks during installation then either user has to investigate and fix issue or wait till expert comes for rescue. I noticed that many time it takes too long to understand root cause of the issue as most of time is spent collecting info about information.

I have created separate branch specific configuration files so that it is easy make available up-to-date and accurate configuration. This also address issue of announcements and educating users. Apart from general usage, developers can consult manual pages for certain flags.

Another use case was to collect sufficient logs. It is very useful to debug. This reduces lot of ping-pong used to happen.

Then to make it more effective, added few auto healing techniques. So known issues used to be fixed automatically. If new issue is encountered, ticket will be filed appropriately. Fix to this new ticket is made as known solution for auto healing.

After few days, keeping hooks up-to-date was a challenge. Then I moved installer script to root folder. Git post commit hook invokes this script. This set up is going well for now.

I have seen most of them know how to use git and find simple to configure git.

I found it simple and effective solution to solve nice problem. It is just an attempt to make happy starting for new engineers with decreased churn.
